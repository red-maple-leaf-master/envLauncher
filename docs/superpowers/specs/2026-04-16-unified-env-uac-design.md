# 统一环境变量提权设计

## 背景

当前项目已经为 JDK 安装链路补充了“仅在写系统环境变量时触发 UAC”的能力，但 Maven 与 Node 仍保留旧行为：

- JDK：非管理员下会主动请求提权，用户取消后不会再静默降级到当前用户环境。
- Maven：写 `MAVEN_HOME` 和 `Path` 时仍会在失败后降级到当前用户环境。
- Node：写 `NODE_HOME` 和多个 `Path` 条目时仍会在失败后降级到当前用户环境，且后续 `npm config` 与 `cnpm` 安装依赖这些环境已配置完成。

这会导致三条安装链路的权限体验不一致，也会让 Maven/Node 在权限不足时给出“安装成功”的误导性结果。

## 目标

将 JDK、Maven、Node 三条安装链路统一为同一套环境变量提权策略：

- 只在写系统环境变量时触发 UAC。
- 下载、解压、目录创建、npm 配置与 `cnpm` 安装保持普通权限。
- 用户取消 UAC 或提权失败时，不再静默降级到当前用户环境。
- 成功、取消、失败三类状态都要有明确日志，并影响控制器的安装完成提示。

## 非目标

- 不修改下载源、安装目录约定和下载/解压流程。
- 不改变 Node 的 `npm config`、`npm install cnpm -g` 的执行策略。
- 不把整个应用提升为管理员运行。
- 不引入新的 UI 交互或对话框，只沿用现有日志输出。

## 推荐方案

采用统一抽象方案：把当前 JDK 的环境变量提权流程提升为通用能力，让 JDK、Maven、Node 共用一套“检测权限 -> 请求提权 -> 写机器级环境 -> 同步当前用户环境 -> 返回明确结果”的服务层流程。

### 方案理由

1. 三条链路的行为、日志和失败语义可以保持一致。
2. 后续如果再新增工具环境变量，只需要接入同一抽象，不需要重复复制 JDK 逻辑。
3. 能把“哪些步骤需要提权、哪些步骤不需要提权”明确固定在服务边界上，避免 Node 的 npm 配置被误提权。

## 架构设计

### 1. 通用环境变量提权结果对象

抽象出通用结果对象，用于描述一次工具环境配置的最终状态：

- `status`：`completed` / `cancelled` / `failed`
- `completed`：是否完成环境配置
- `usedElevation`：是否实际走过 UAC 提权
- `message`：面向日志与控制器的最终描述

JDK、Maven、Node 都返回同一语义结构，控制器不再通过“是否抛异常”来推断安装是否真正完成。

### 2. 通用提权环境服务模板

在环境服务层收敛成统一流程：

1. 根据工具类型构造需要写入的环境变量和值。
2. 根据工具类型计算 PATH 的目标内容。
3. 如果当前进程已是管理员，则直接写机器级环境变量并同步当前用户环境。
4. 如果当前不是管理员，则调用统一的 UAC 执行器。
5. UAC 成功后，同步当前用户环境，保持当前用户可见性一致。
6. UAC 取消或失败时，返回不完整结果，不写任何用户级回退值。

这层模板负责权限策略和结果语义；各具体服务只负责提供自己的变量名与 PATH 条目。

### 3. 工具差异收敛点

JDK：

- 变量：`JAVA_HOME`
- PATH 条目：`%JAVA_HOME%\bin`

Maven：

- 变量：`MAVEN_HOME`
- PATH 条目：`<mavenHome>\bin`

Node：

- 变量：`NODE_HOME`
- PATH 条目：`<nodeHome>`、`<nodeHome>\node_global`、`<nodeHome>\node_cache`
- 仅环境变量部分使用统一提权流程
- `configureNpmPaths(...)` 仍在普通权限下执行，且只有环境变量成功完成后才继续执行

## 行为设计

### JDK

保持现有行为不变，只把内部实现迁移到统一抽象上，避免以后与 Maven/Node 再次漂移。

### Maven

安装文件完成后：

- 设置 `MAVEN_HOME` 和系统 `Path` 时检测管理员权限。
- 非管理员下主动弹出 UAC。
- 用户同意后写系统环境变量，并同步当前用户环境。
- 用户取消或提权失败时，明确记录“Maven 文件已安装，但环境变量未更新”。

### Node

安装文件和目录创建完成后：

- 仅在设置 `NODE_HOME` 和相关 `Path` 时检测管理员权限。
- 非管理员下主动弹出 UAC。
- 用户同意后写系统环境变量，并同步当前用户环境。
- 只有环境变量配置完成，才继续执行 `npm config set ...` 与 `npm install cnpm -g`。
- 用户取消或提权失败时，Node 安装不应继续执行 npm 配置步骤，因为 PATH 和 `NODE_HOME` 前置条件未满足。

## 错误处理与日志

所有工具统一记录以下日志节点：

- 开始请求管理员权限
- 管理员权限批准并完成系统环境写入
- 用户取消管理员权限
- 管理员环境写入失败
- 文件已安装但环境变量未完成

禁止的旧行为：

- 系统环境写入失败后自动回退成当前用户环境并继续报成功
- 用户取消 UAC 后仍继续输出完整安装完成日志

Node 额外要求：

- 如果环境变量阶段未完成，必须明确记录 npm 配置被跳过
- npm 配置失败仍保留现有异常日志，但这属于普通权限阶段，不触发新的提权

## 控制器行为

控制器统一消费通用结果对象：

- `completed=true`：输出对应工具安装完成日志
- `completed=false`：输出“文件已安装，但环境配置未完成”的日志

Node 控制器链路需要额外依赖服务层结果，确保环境变量未完成时不会进入 npm 配置成功分支。

## 测试设计

新增或扩展测试覆盖：

1. Maven 非管理员场景下，请求提权成功后返回完成状态。
2. Maven 用户取消 UAC 时，不回退到用户级环境，并返回不完整状态。
3. Node 非管理员场景下，请求提权成功后才继续 npm 配置。
4. Node 用户取消 UAC 时，不回退到用户级环境，且不会继续执行 npm 配置。
5. JDK 既有行为保持通过，作为统一抽象迁移后的回归保障。

验证命令保持项目约定：

- `mvn clean compile`
- `mvn test`

## 风险与控制

### 风险 1：统一抽象时误伤 JDK 已有行为

控制方式：

- 先保留 JDK 现有结果语义，再逐步把 Maven/Node 接入同一流程。
- 保留 JDK 回归测试，避免抽象过程中行为回退。

### 风险 2：Node 在环境变量失败后仍继续执行 npm 配置

控制方式：

- 让 Node 服务先拿到统一环境配置结果，再决定是否继续 `configureNpmPaths(...)`。
- 测试覆盖“取消 UAC 时 npm 配置不会执行”。

### 风险 3：控制器日志再次出现假成功

控制方式：

- 控制器只依据结果对象判定是否真正完成。
- 不再用“任务没抛异常”当作安装完成标准。

## 实施顺序

1. 提炼通用环境变量提权结果对象与模板流程。
2. 迁移 JDK 到统一抽象，保持现有测试通过。
3. 接入 Maven。
4. 接入 Node，并把 npm 配置与环境变量成功结果绑定。
5. 补齐回归测试并执行编译、测试验证。
