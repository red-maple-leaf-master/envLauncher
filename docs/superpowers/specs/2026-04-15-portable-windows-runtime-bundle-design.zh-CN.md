# 绿色版内置运行时打包设计

## 背景

`env-launcher` 是一个面向 Windows 的 JavaFX 桌面应用，当前基于 JDK 17 和 Maven 构建。
项目已经接入了 `javafx-maven-plugin`，但目前打包说明仍然比较通用，还不能明确保证最终用户在没有预装 JDK/JRE 的机器上直接运行。

这次新增目标是产出一个 Windows 绿色版发布目录，满足以下要求：

- 包含应用启动程序
- 包含内置 Java 运行时
- 解压后可直接双击 `.exe` 运行
- 目标机器不需要单独安装 JDK 或 JRE

本次改动必须保持 Java 17 基线不变，并继续使用 Maven 构建。

## 推荐方案

基于现有 `javafx-maven-plugin` 生成 `jlink` 运行时镜像目录。

推荐原因：

- 与当前 JavaFX 模块化项目结构一致
- 可以直接产出包含内置运行时的绿色目录
- 不需要引入第三方 exe 包装器或额外安装器工具链
- 改动范围可以控制在构建配置和文档层面

不采用的方案：

- `jpackage` app-image：可行，但相对当前“绿色目录”目标来说配置更重，当前阶段没有必要
- `jar + 外置 JRE + 自定义启动包装`：维护成本更高，也不如当前模块化 JavaFX 项目匹配

## 产物设计

构建后应在 `target/` 下生成一个 Windows 绿色版目录，目录中至少包含：

- `bin/` 下的启动 `.exe`
- 内置 runtime 目录
- 应用模块及启动所需依赖

最终用户使用流程：

1. 在有 JDK 17 的构建机上执行打包命令。
2. 从 `target/` 取出生成的绿色版目录。
3. 将该目录直接拷贝或压缩后分发。
4. 在另一台 Windows 机器上解压后，直接双击其中的 `.exe` 运行。

该产物明确不是安装包，而是一个自包含的绿色运行目录。

## 构建配置调整

将更新 `pom.xml`，使 JavaFX 插件配置明确支持绿色版运行时目录生成。

计划调整内容：

- 保持 Java 17 与当前 JavaFX 版本不变
- 保持 `winLauncherType=gui`
- 固定 launcher 名称，保证对 Windows 用户可读
- 固定运行镜像输出名，保证和项目名一致
- 在文档中明确生成绿色目录的 Maven 命令

优先推荐的打包命令：

```bash
mvn clean javafx:jlink
```

如果可以在不增加额外副作用的前提下，把 `package` 阶段也安全接到同样的产物链路上，可以在实现阶段再评估。当前默认建议仍然是显式执行 `javafx:jlink`，因为更清晰，也更稳妥。

## 运行时与资源行为

内置运行时必须覆盖当前模块化应用在 Windows 上正常启动所需内容：

- Java 基础运行时模块
- 应用当前使用到的 JavaFX 模块
- 项目已经依赖的 XML 模块支持

本次不引入应用功能行为变化，只调整应用的打包与启动方式。

资源解析要求保持不变：

- FXML 和样式资源在打包后仍可正常加载
- 图标资源在 launcher 和窗口中仍可正常使用

## 验证方案

只有满足以下条件，才可以认为该改动完成：

1. `mvn clean compile` 成功
2. `mvn test` 成功
3. `mvn clean javafx:jlink` 成功
4. 生成的绿色版目录中存在 Windows 启动 `.exe`
5. 生成的绿色版目录中存在内置 runtime
6. 在构建机上从生成目录直接启动 launcher 时，不依赖系统 `PATH` 中的 `java`

第一阶段实现时，不强制要求在第二台纯净 Windows 机器上验证，但这是后续发布前建议补做的检查。

## 风险与应对

风险：插件生成的目录名或 launcher 名称不适合发布使用。
应对：在 Maven 配置中显式指定镜像名和 launcher 名，并在文档中写清最终输出路径。

风险：运行镜像缺少某个启动必需模块。
应对：保持与 `module-info.java` 一致，并从生成后的绿色目录直接启动验证。

风险：文档仍然让用户误以为必须先安装 JDK/JRE。
应对：更新 README 中的打包与运行说明，区分“开发构建需要 JDK”与“最终用户运行不需要 JDK/JRE”。
