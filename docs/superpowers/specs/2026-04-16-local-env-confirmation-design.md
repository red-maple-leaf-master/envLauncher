# 本地环境导入二段式确认设计

日期：2026-04-16

## 背景

当前本地环境导入入口在用户选择完本地目录后，会立刻开始执行环境变量设置。
这条交互路径缺少明显的确认步骤，容易导致用户误选目录后直接修改系统环境变量。

本次设计目标是把“选择本地目录”和“应用本地环境”拆成两个明确步骤，让用户先看到已选路径，再决定是否执行环境设置。

## 目标

1. 选择本地 JDK、Maven、Node 目录后，不立即执行环境变量设置。
2. 主界面显示每个工具当前已选的本地路径。
3. 只有用户点击单独的“应用本地环境”按钮时，才执行环境变量设置。
4. 保持现有下载并安装流程不变。

## 非目标

1. 不改动现有下载、解压、下载源逻辑。
2. 不增加自动扫描本机环境目录。
3. 不引入额外弹窗确认框，确认步骤放在主界面完成。

## 交互设计

### 每个工具新增二段式本地导入控件

对 JDK、Maven、Node 各自增加三类控件：

1. `Select Local JDK/Maven/Node`
2. 只读路径显示框
3. `Apply Local JDK/Maven/Node`

路径显示框用于展示当前已选本地目录，默认为空，不可手工编辑。

### 两段式流程

第一步：选择本地目录

1. 点击 `Select Local ...`
2. 打开 `DirectoryChooser`
3. 选择完成后先做目录合法性校验
4. 校验通过则只更新路径显示框并输出日志
5. 不触发环境变量设置

第二步：应用本地环境

1. 点击 `Apply Local ...`
2. 读取当前路径显示框内容
3. 调用现有环境变量设置逻辑
4. 执行期间进入 `busy`
5. 完成、失败或取消后恢复可操作状态

## 路径显示规则

### JDK

显示框保存 JDK 根目录，例如：

- `D:\environment\JDK\jdk-17.0.2`

### Maven

显示框保存 Maven 根目录，例如：

- `D:\environment\apache-maven-3.9.10`

### Node

显示框保存 Node 根目录，例如：

- `D:\environment\node-v20.19.2-win-x64`

## 按钮启用规则

### Select Local

1. 非 `busy` 时可点击
2. `busy` 时禁用

### Apply Local

1. 对应路径显示框非空且非 `busy` 时可点击
2. 对应路径为空时禁用
3. `busy` 时禁用

### 原有安装按钮

继续保持当前规则：

1. 依赖安装目录
2. `busy` 时禁用

## 控制器设计

在 `EnvInstallerController` 中增加：

1. 三个只读本地路径显示框字段
2. 三个本地目录选择入口
3. 三个本地环境应用入口
4. 本地路径清理与读取辅助逻辑

### 选择本地目录方法职责

1. 打开目录选择器
2. 校验目录是否合法
3. 合法则写入对应路径显示框
4. 非法则保持原值不变
5. 不调用环境变量设置逻辑

### 应用本地环境方法职责

1. 读取路径显示框
2. 若为空则提示先选择本地目录
3. 若不为空则调用现有环境变量设置逻辑
4. 复用当前 JDK/Maven/Node 的环境变量服务

## 日志设计

### 选择阶段

至少记录：

1. 用户开始选择本地目录
2. 用户取消选择
3. 目录合法并已缓存到界面
4. 目录非法

### 应用阶段

至少记录：

1. 开始应用本地环境
2. 环境变量设置成功
3. 环境变量设置失败
4. 管理员授权取消

## 状态约束

### 选择本地目录

不进入 `busy`。

原因：

1. 只是本地交互与校验
2. 无后台长任务

### 应用本地环境

进入 `busy`。

原因：

1. 会触发后台 `Task`
2. 可能涉及 UAC 提权与注册表写入

## 测试方案

### 控制器测试

覆盖以下行为：

1. 选择本地目录只更新路径，不触发环境配置
2. 路径为空时 `Apply Local ...` 不可用或直接拒绝执行
3. 路径存在时 `Apply Local ...` 才能触发环境配置
4. `busy` 时选择和应用按钮都禁用

### 回归测试

确保以下行为保持不变：

1. JDK 下载后自动配置环境
2. Maven 下载后自动配置环境
3. Node 下载后自动配置环境
4. 原有环境变量提权逻辑不变

## 风险与取舍

### 风险

1. 主界面会变高，需要注意布局换行和可读性
2. 新增路径显示框后，控件状态组合会变多

### 取舍

本次选择主界面二段式确认，而不是弹确认框。

原因：

1. 已选路径长期可见，状态更明确
2. 不依赖临时弹窗记忆
3. 用户可以先检查路径，再执行应用
4. 与现有主界面操作风格更一致
